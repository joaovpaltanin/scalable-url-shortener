# V5 — Replication + Eventual Consistency

[← Back to overview](../../README.md)

## Problem

V4 introduced sharding, so no single PostgreSQL instance is the write bottleneck. But each shard is still a single node:

1. **No read replicas.** Cache misses and all reads/writes for a shard hit the same primary.
2. **Shard failure hurts availability.** If shard 1 dies, codes routed to shard 1 become unavailable.
3. **Eventual consistency is unavoidable once you replicate.** Reads from replicas may be stale (replication lag).

V5 adds **one read replica per shard**. Writes go to the shard primary, reads can go to the replica, and the system explicitly handles **eventual consistency**.

## Replication Strategy

Each shard has:

- **Primary**: handles writes (`INSERT`) and is the source of truth
- **Replica**: receives changes asynchronously from the primary (streaming replication)

In V5, we keep the shard key the same (the `code`) and add **read/write split** inside each shard.

## Read/Write Routing

### Write path (shorten)

Unchanged from V4:

1. Generate `code`
2. Route to the shard primary using the shard key
3. Insert into primary
4. Write-through to Redis

Because we write-through to Redis, most subsequent reads will never touch the database.

### Read path (resolve) and eventual consistency

1. Check Redis for `url:{code}`
2. If cache hit: return immediately
3. If cache miss:
   - Query the **replica** for that shard first
   - If replica returns nothing, fall back to the **primary**

That fallback is the critical piece: replicas can be behind (replication lag). Without it, a newly created code could return **404** for a short window if the cache is cold and the read goes to a stale replica.

This is **eventual consistency**: the system will converge, but reads may briefly observe stale state.

## What changed in code

- **Config**: each shard now has `writeUrl` and `readUrl`
- **ShardRouter**: exposes `writeTemplateFor(code)` and `readTemplateFor(code)`
- **ShardedUrlRepository**:
  - `save()` uses the **primary**
  - `findByCode()` reads from **replica**, then falls back to **primary** on miss (replication lag guard)

## Where it hurts

V5 improves read scalability and (for cache-miss reads) reduces load on primaries, but:

1. **Replication lag exists.** You must design around stale reads.
2. **Failover is not automatic.** Stopping a primary breaks writes for that shard unless you promote a replica.
3. **Operational overhead increases.** Twice the Postgres nodes to monitor, backup, and upgrade.
