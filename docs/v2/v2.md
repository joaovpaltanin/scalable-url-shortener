# V2 — Stateless App + Load Balancer

[← Back to overview](../../README.md)

## Problem

V1 proved two things:

1. **There's a throughput ceiling.** A single app instance handles ~9,100 req/s at best. Past that, latency explodes with no way to add capacity without downtime.
2. **One container down = total outage.** The chaos test showed that killing the single app process causes 100% of requests to fail — no failover, no recovery.

V2 solves both by running **multiple stateless replicas** behind an **Nginx load balancer**. If one dies, traffic routes to the others automatically.

## Architecture

```text
                          ┌──────────────────┐
                          │   Nginx (LB)     │
                          │     :80          │
                          └──────┬───────────┘
                     ┌───────────┼───────────┐
                     ▼           ▼           ▼
              ┌───────────┐ ┌───────────┐ ┌───────────┐
              │  App (1)  │ │  App (2)  │ │  App (3)  │
              │   :8080   │ │   :8080   │ │   :8080   │
              └─────┬─────┘ └─────┬─────┘ └─────┬─────┘
                    └─────────────┼─────────────┘
                                  ▼
                          ┌──────────────┐
                          │  PostgreSQL  │
                          │    :5432     │
                          └──────────────┘
```

**What changed from V1:**

| Component | V1 | V2 |
|-----------|----|----|
| App instances | 1 | 3 (via `deploy.replicas`) |
| Load balancer | None | Nginx (round-robin) |
| Exposed port | 8080 (direct to app) | 80 (via Nginx) |
| Failover | None — total outage | Automatic — `proxy_next_upstream` |
| DB | Single PostgreSQL | Same (unchanged) |

## Why No Code Changes?

The app is already stateless:

- No HTTP sessions or sticky state
- No local caches or in-memory data
- Code generation uses `SecureRandom` (no shared counter)
- All state lives in PostgreSQL

The only thing that matters is the `Host` header — `UrlController` uses it to build the short URL. Nginx forwards the original `Host` header, so short URLs stay correct.

## API

Unchanged from V1. The port changes from `8080` to `80`.

| Method | Endpoint       | Description                    |
|--------|----------------|--------------------------------|
| POST   | `/api/shorten` | Create a short URL             |
| GET    | `/r/{code}`    | Redirect to original URL (302) |

**Create:**

```bash
curl -X POST http://localhost/api/shorten \
  -H "Content-Type: application/json" \
  -d '{"url": "https://example.com/very-long-path"}'
```

```json
{ "shortUrl": "http://localhost/r/aBcD3fG" }
```

**Redirect:**

```bash
curl -v http://localhost/r/aBcD3fG
# → 302 Found, Location: https://example.com/very-long-path
```

## How to Run

**Prerequisites:** Docker and Docker Compose installed.

```bash
# Start everything (Postgres + 3 App replicas + Nginx)
docker compose up --build

# App is available at http://localhost (port 80)
```

To stop and clean up:

```bash
docker compose down -v
```

## Load Testing

### Baseline (200 VUs)

```bash
docker compose run --rm k6 run v2/baseline.js
```

Custom parameters:

```bash
docker compose run --rm k6 run -e SEED_COUNT=1000 v2/baseline.js
```

### Chaos Test (kill a replica mid-traffic)

```bash
# Terminal 1 — start the chaos test
docker compose run --rm k6 run v2/chaos.js

# Terminal 2 — while k6 is running, kill one replica
docker compose scale app=2

# Watch k6 output: error rate should stay near 0%
# Then restore:
docker compose scale app=3
```

Steady 100 VUs for 60 seconds. The goal: prove that **one replica down ≠ outage**. Nginx detects the failure and routes traffic to the surviving replicas.

## Results

### Baseline (200 VUs)

| Metric             | V1          | V2          |
|--------------------|-------------|-------------|
| Throughput (req/s) | 1,125       | 1,112.8     |
| Latency p95        | 4.08 ms     | 4.1 ms      |
| Error rate         | 0.00%       | 0.00%       |
| Total requests     | 170,991     | 169,177     |

> Throughput and latency are nearly identical to V1 -- expected, since the bottleneck is PostgreSQL and the Nginx proxy adds negligible overhead. The important thing is that the system handles the same load correctly with 3 replicas behind a load balancer. The real win shows up in the chaos test below.

### Chaos (kill one replica mid-traffic)

| Metric           | V1 (kill app)      | V2 (kill 1 of 3)   |
|------------------|--------------------|--------------------|
| Total requests   | 3,018              | 55,510             |
| Successes        | 2,000              | 55,010             |
| Failures         | 518                | 0                  |
| Error rate       | 17.16%             | 0.00%              |
| Availability     | **0%** during kill | **100%**           |
| Throughput       | 47.5 req/s         | 601.3 req/s        |

> Three replicas running, one killed mid-test with `docker compose scale app=2`. V1 had **100% failure rate** the moment the single instance died -- 518 failed requests in ~10 seconds of downtime. V2 with one replica killed mid-traffic served 55,510 requests with **zero failures**. Nginx detected the dead instance and routed all traffic to the surviving two -- no errors, no downtime, no manual intervention.

## Where It Hurts

V2 solves horizontal scaling and single point of failure. But new bottlenecks remain:

**1. Every read still hits the database.**
With 3 app replicas all hammering PostgreSQL, the DB becomes the bottleneck sooner. More replicas means more connections, more queries, more pressure on a single Postgres instance.

**2. The database is still a single point of failure.**
Kill Postgres and everything dies — no amount of app replicas helps. The DB has no replicas, no failover.

**3. No caching.**
Every `GET /r/{code}` does a `SELECT` query. Short URLs are read-heavy and rarely change — perfect for caching.

## What's Next: V3

Every redirect still goes to PostgreSQL. With 3 replicas generating 3x the query load, the DB ceiling arrives faster. V3 adds **Redis** as a cache layer — short codes are immutable once created, making them ideal cache candidates. Reads hit Redis first, falling back to Postgres only on cache miss.
