# V4 — Database Sharding

[← Back to overview](../../README.md)

## Problem

V3 proved that Redis absorbs read load — most redirects never touch PostgreSQL. But the **write path is still a bottleneck**:

1. **Every `POST /api/shorten` hits a single database.** All 3 app replicas funnel writes into one PostgreSQL instance. As write volume grows, that single DB becomes the ceiling.
2. **The database is still a single point of failure.** Kill Postgres and everything dies — Redis only caches what it has seen. New writes and cache misses still need Postgres.

V4 introduces **sharding** — distributing data across **2 PostgreSQL instances** using deterministic hash routing. Writes spread across both shards, doubling write capacity. And if one shard dies, the other keeps serving its codes.

> **New to sharding?** Read [Sharding in Software Engineering](sharding.md) — what it is, why and when to use it, routing strategies, and trade-offs.

```text
              ┌───────────┐ ┌───────────┐ ┌───────────┐

              │   :8080   │ │   :8080   │ │   :8080   │
              └─────┬─────┘ └─────┬─────┘ └─────┬─────┘
                    └─────────────┼─────────────┘
                    ┌─────────────┼─────────────┐
                    ▼             ▼             ▼
             ┌────────────┐ ┌──────────┐ ┌──────────┐
             │   Redis    │ │ Shard 0  │ │ Shard 1  │
             │   :6379   │ │  :5432   │ │  :5433   │
             │  (cache)  │ │ (Postgres)│ │ (Postgres)│
             └────────────┘ └──────────┘ └──────────┘
```

**What changed from V3:**

| Component | V3 | V4 |
|-----------|----|----|
| Database | 1 PostgreSQL | 2 PostgreSQL shards |
| Write path | App → PostgreSQL → Redis | App → ShardRouter → Shard N → Redis |
| Read path (miss) | App → Redis → PostgreSQL | App → Redis → ShardRouter → Shard N |
| Routing | None — single DB | `Math.abs(code.hashCode() % 2)` |
| Repository | Spring Data JPA (`UrlRepository`) | `JdbcTemplate` (`ShardedUrlRepository`) |
| DB failure | Total outage | ~50% of codes still resolve |

## Sharding Strategy

**Deterministic hash routing:** `shardIndex = Math.abs(code.hashCode() % 2)`

The short code is the shard key. Once generated, it always maps to the same shard — no lookup table, no coordination between shards.

**Write path (shorten):**
1. Generate a random 7-char code
2. Compute `shardIndex = Math.abs(code.hashCode() % 2)`
3. INSERT into the target shard's `shortened_urls` table
4. Write to Redis (cache-through)

**Read path (resolve):**
1. Check Redis for `url:{code}`
2. If **hit**: return immediately
3. If **miss**: compute the shard from the code, query only that shard, populate Redis

**Why `hashCode() % N`?** The codes are randomly generated from `[a-zA-Z0-9]`, so `String.hashCode()` distributes them uniformly across shards. `Math.abs()` handles the case where `hashCode()` returns a negative integer (Java's `%` preserves sign).

**Why drop JPA?** Spring Data JPA assumes a single `DataSource`. With 2 shards, we need explicit control over which `DataSource` receives each query. `JdbcTemplate` gives us that control with minimal complexity.

## Code Changes

**New:** `ShardingDataSourceConfig` — `@ConfigurationProperties` that creates a `DataSource` + `JdbcTemplate` for each shard from `application.yml` config.

**New:** `ShardRouter` — deterministic routing: `shardFor(code)` returns the shard index, `templateFor(code)` returns the `JdbcTemplate` for that shard.

**New:** `ShardedUrlRepository` — replaces `UrlRepository`. Uses `JdbcTemplate` directly: `save(code, url)`, `findByCode(code)`, `existsByCode(code)`. Each method routes to the correct shard via `ShardRouter`.

**Modified:** `UrlService` — swapped `UrlRepository` for `ShardedUrlRepository`. The `shorten()` and `resolve()` methods are structurally identical; only the repository dependency changed. Catches `DuplicateKeyException` instead of `DataIntegrityViolationException`.

**Removed:** `UrlRepository` (JPA interface) — no longer needed without JPA.

**Modified:** `ShortenedUrl` — stripped JPA annotations (`@Entity`, `@Table`, `@Id`, etc.), kept as a plain domain class.

## API

Unchanged from V3. Same port, same endpoints.

| Method | Endpoint       | Description                    |
|--------|----------------|--------------------------------|
| POST   | `/api/shorten` | Create a short URL             |
| GET    | `/r/{code}`    | Redirect to original URL (302) |

**Create:**

```bash
curl -X POST http://localhost/api/shorten \
  -H "Content-Type: application/json" \
  -d '{"url": "https://example.com/very-long-path"}'
```

```json
{ "shortUrl": "http://localhost/r/aBcD3fG" }
```

**Redirect:**

```bash
curl -v http://localhost/r/aBcD3fG
# → 302 Found, Location: https://example.com/very-long-path
```

## How to Run

**Prerequisites:** Docker and Docker Compose installed.

```bash
# Start everything (2 Postgres shards + Redis + 3 App replicas + Nginx)
docker compose up --build

# App is available at http://localhost (port 80)
```

To stop and clean up:

```bash
docker compose down -v
```

## Load Testing

### Baseline (200 VUs)

```bash
docker compose run --rm k6 run v4/baseline.js
```

Custom parameters:

```bash
docker compose run --rm k6 run -e SEED_COUNT=1000 v4/baseline.js
```

### Chaos Test (kill one shard mid-traffic)

```bash
# Terminal 1 — start the chaos test
docker compose run --rm k6 run v4/chaos.js

# Terminal 2 — while k6 is running, kill one shard
docker compose stop postgres-1

# Watch k6 output: writes to the dead shard fail,
# writes to the alive shard succeed, reads hit Redis.
# Then restore:
docker compose start postgres-1
```

Steady 100 VUs for 60 seconds, **70/30 read/write mix**. The write mix is critical — reads hit Redis (cache-through from seeding), so they succeed even with a shard down. Writes must hit Postgres directly, so writes routed to the dead shard fail while writes to the alive shard succeed. The goal: prove that **one shard down ≠ total outage**.

## Results

### Baseline (200 VUs)

| Metric             | V3          | V4          |
|--------------------|-------------|-------------|
| Throughput (req/s) | 1,091.0     | 1,093.1     |
| Latency p95        | 5.5 ms      | 4.1 ms      |
| Redirect p95       | 3.5 ms      | 2.9 ms      |
| Error rate         | 0.00%       | 0.00%       |
| Total requests     | 168,860     | 169,495     |

> Throughput is comparable to V3 — expected, since the 90/10 read/write mix means most requests hit Redis regardless of sharding. The key improvements are in latency: p95 dropped from 5.5 ms to 4.1 ms, and redirect p95 from 3.5 ms to 2.9 ms. With writes split across 2 shards, each Postgres instance handles ~8,400 writes instead of ~16,800, reducing connection contention and query latency. Zero errors across 169,495 requests. The real win shows up under write-heavy load or when scaling beyond what a single DB can handle.

### Chaos (kill one shard mid-traffic)

| Metric           | V3 (kill Redis)    | V4 (kill shard 1)  |
|------------------|--------------------|--------------------|
| Total requests   | 12,006             | 30,444             |
| Successes        | 11,406             | 29,832             |
| Failures         | 100                | 111                |
| Error rate       | 0.83%              | 0.36%              |
| Throughput       | 149.7 req/s        | 441.9 req/s        |

> Shard 1 was killed mid-test with `docker compose stop postgres-1`. The 70/30 read/write mix exposed the failure: 111 writes routed to the dead shard failed, while writes to shard 0 and all reads (served from Redis cache) continued normally — 0.36% error rate. Compare with V3: killing Redis caused 0.83% errors and throughput collapsed to 149.7 req/s because every read fell back to Postgres. Here, throughput stayed at 441.9 req/s and the vast majority of traffic was unaffected. The key takeaway: **one shard down = partial write degradation, not total outage**.

## Where It Hurts

V4 distributes writes across 2 shards — no single database is the bottleneck. But new problems remain:

**1. No read replicas.**
Cache misses and all writes for a given shard go to a single PostgreSQL instance. There's no way to distribute read load within a shard.

**2. Shard failure = partial data loss.**
When one shard dies, all codes routed to it become unavailable. There's no replica to failover to within a shard.

**3. No cross-shard queries.**
Operations that need to scan all data (e.g., analytics, bulk export) require querying every shard and merging results. The current design doesn't support this.

## What's Next: V5

Writes are now distributed, but each shard is still a single instance — no replicas, no failover within a shard. V5 introduces **replication** — each shard gets a read replica, so cache misses can be served by replicas instead of the primary, and a shard failure doesn't mean total data unavailability.
