# V3 — Redis Cache Layer

[← Back to overview](../../README.md)

## Problem

V2 proved that 3 stateless replicas behind Nginx survive partial failures with zero errors. But throughput didn't improve — the bottleneck is PostgreSQL, and every redirect still does a `SELECT` query:

1. **Every `GET /r/{code}` hits the database.** With 3 replicas generating 3x the query load, the DB ceiling arrives faster. Under high read traffic, PostgreSQL connection pool saturation becomes the limiting factor.
2. **Short URLs are immutable.** Once created, a code-to-URL mapping never changes. This is the textbook use case for caching — high read frequency, zero writes after creation.

V3 adds **Redis** as a cache layer. Reads hit Redis first (sub-millisecond), falling back to Postgres only on cache miss. Writes populate the cache immediately so subsequent reads never miss.

## Architecture

```text
                          ┌──────────────────┐
                          │   Nginx (LB)     │
                          │     :80          │
                          └──────┬───────────┘
                     ┌───────────┼───────────┐
                     ▼           ▼           ▼
              ┌───────────┐ ┌───────────┐ ┌───────────┐
              │  App (1)  │ │  App (2)  │ │  App (3)  │
              │   :8080   │ │   :8080   │ │   :8080   │
              └─────┬─────┘ └─────┬─────┘ └─────┬─────┘
                    └─────────────┼─────────────┘
                           ┌──────┴──────┐
                           ▼             ▼
                    ┌────────────┐ ┌──────────────┐
                    │   Redis    │ │  PostgreSQL  │
                    │   :6379   │ │    :5432     │
                    │  (cache)  │ │   (source)   │
                    └────────────┘ └──────────────┘
```

**What changed from V2:**

| Component | V2 | V3 |
|-----------|----|----|
| Cache | None — every read hits DB | Redis (read-through + write-through) |
| Read path | App → PostgreSQL | App → Redis → (miss) → PostgreSQL |
| Write path | App → PostgreSQL | App → PostgreSQL → Redis |
| Failure mode (cache down) | N/A | Graceful fallback to DB |
| DB query load | 100% of reads | Only cache misses |

## Caching Strategy

**Read-through (resolve):**
1. Check Redis for key `url:{code}`
2. If **hit**: return the cached URL immediately
3. If **miss**: query PostgreSQL, store result in Redis, return

**Write-through (shorten):**
1. Save to PostgreSQL (source of truth)
2. Write to Redis immediately after successful save
3. Subsequent reads for this code will always hit cache

**Why no TTL?** Short URL mappings are immutable — once created, `code → originalUrl` never changes. No TTL means no cache misses after the first access, and no stale data risk.

**Why no eviction concern?** Each cache entry is a 7-char key + a URL string. Even with millions of entries, the memory footprint is negligible for Redis.

**Graceful degradation:** If Redis is unavailable (network error, crash), the app catches the exception and falls back to PostgreSQL transparently. Redis is a performance layer, not a dependency.

## Code Changes

The app code changes are minimal — the controller and API remain identical.

**New:** `UrlCacheService` — wraps `StringRedisTemplate` with `get(code)` and `put(code, url)`. All Redis calls are wrapped in try-catch to ensure graceful fallback.

**Modified:** `UrlService`
- `resolve()` checks cache before DB, populates cache on miss
- `shorten()` writes to cache after successful DB save

## API

Unchanged from V2. Same port, same endpoints.

| Method | Endpoint       | Description                    |
|--------|----------------|--------------------------------|
| POST   | `/api/shorten` | Create a short URL             |
| GET    | `/r/{code}`    | Redirect to original URL (302) |

**Create:**

```bash
curl -X POST http://localhost/api/shorten \
  -H "Content-Type: application/json" \
  -d '{"url": "https://example.com/very-long-path"}'
```

```json
{ "shortUrl": "http://localhost/r/aBcD3fG" }
```

**Redirect:**

```bash
curl -v http://localhost/r/aBcD3fG
# → 302 Found, Location: https://example.com/very-long-path
```

## How to Run

**Prerequisites:** Docker and Docker Compose installed.

```bash
# Start everything (Postgres + Redis + 3 App replicas + Nginx)
docker compose up --build

# App is available at http://localhost (port 80)
```

To stop and clean up:

```bash
docker compose down -v
```

## Load Testing

### Baseline (200 VUs)

```bash
docker compose run --rm k6 run v3/baseline.js
```

Custom parameters:

```bash
docker compose run --rm k6 run -e SEED_COUNT=1000 v3/baseline.js
```

### Chaos Test (kill Redis mid-traffic)

```bash
# Terminal 1 — start the chaos test
docker compose run --rm k6 run v3/chaos.js

# Terminal 2 — while k6 is running, kill Redis
docker compose stop redis

# Watch k6 output: error rate should stay near 0%
# The app falls back to PostgreSQL — higher latency, but no errors.
# Then restore:
docker compose start redis
```

Steady 100 VUs for 60 seconds. The goal: prove that **Redis down ≠ outage**. The app catches Redis errors and falls back to PostgreSQL transparently.

## Results

### Baseline (200 VUs)

| Metric             | V2          | V3          |
|--------------------|-------------|-------------|
| Throughput (req/s) | 1,112.8     | 1,091.0     |
| Latency p95        | 4.1 ms      | 5.5 ms      |
| Redirect p95       | —           | 3.5 ms      |
| Error rate         | 0.00%       | 0.00%       |
| Total requests     | 169,177     | 168,860     |

> Overall throughput and latency are comparable to V2 — the bottleneck is still PostgreSQL for writes. The key metric is **Redirect p95 at 3.5 ms**: reads that hit Redis skip the database entirely. The 90/10 read/write split means 151,925 reads were served from cache, with only 16,935 writes touching Postgres. Zero errors across the full run.

### Chaos (kill Redis mid-traffic)

| Metric           | V2 (kill replica) | V3 (kill Redis)    |
|------------------|-------------------|--------------------|
| Total requests   | 55,510            | 12,006             |
| Successes        | 55,010            | 11,406             |
| Failures         | 0                 | 100                |
| Error rate       | 0.00%             | 0.83%              |
| Throughput       | 601.3 req/s       | 149.7 req/s        |

> Redis was killed mid-test with `docker compose stop redis`. The app detected Redis failures and fell back to PostgreSQL — 100 requests timed out during the transition (~0.83% error rate), but the system never went down. Throughput dropped from V2's 601 req/s to 149.7 req/s because every read now hits Postgres instead of cache, and the Redis connection timeout adds latency during the failover window. The key takeaway: **Redis down ≠ outage**. It's a performance layer, not a dependency.

## Where It Hurts

V3 solves the read amplification problem — most reads never touch PostgreSQL. But new bottlenecks remain:

**1. The database is still a single point of failure.**
Kill Postgres and everything dies. Redis only caches what it has seen — new writes and cache misses still need Postgres.

**2. Writes still hit a single database.**
Every `POST /api/shorten` goes to one PostgreSQL instance. As write volume grows, the single DB becomes the bottleneck.

**3. No read replicas.**
Cache misses and all writes go to the same primary. There's no way to distribute read load across multiple DB instances.

## What's Next: V4

The cache absorbs reads, but writes still funnel into a single PostgreSQL instance. As the URL table grows and write volume increases, the single DB becomes the ceiling. V4 introduces **sharding** — distributing writes across multiple database instances so no single node is the bottleneck.
